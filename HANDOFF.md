# Companion 交接文档

## 1. 当前架构结论
- 目标架构：Python 负责核心计算；Web/Java 作为壳层展示与转发。
- 当前进度：聊天主链路与记忆主链路已迁到 Python；主动调度仍主要在前端。

## 2. 已完成的关键工作

### 2.1 聊天与记忆核心下沉到 Python
- Web 通过 `/core/chat/reply` 调用核心聊天。
- Python 在主链路中完成：会话获取、历史拼接、LLM 回复、消息落库、记忆更新。
- 新增/完善接口：
  - `POST /chat/history`
  - `POST /chat/reply`

### 2.2 本地持久化落地
- 聊天记录与记忆都已落地到本地 SQLite。
- 默认数据库路径：`data/companion.db`（未设置 `SQLITE_DB_PATH` 时启用）。
- 已验证：服务重启后，历史与记忆可正常加载。

### 2.3 稳定性修复（500/502）
- 上游超时/HTTP 异常加了兜底，避免未捕获 500。
- 提高了 DeepSeek 客户端超时设置。
- `web-shell` 代理改为多线程并补充超时返回（504），避免单请求阻塞拖垮。
- 前端 `send()` 已分离：主聊天失败与后处理失败不再混淆。

### 2.4 web-shell 冗余清理
- 已删除旧的未引用模块与旧代理脚本，降低结构噪音。
- 保留当前可运行所需的 `web-shell/index.html` 与 `web-shell/server.py`。

## 3. 近期问题复盘

### 3.1 频繁失败的根因
- 一部分来自上游模型超时或鉴权失败（401/504/502）。
- 更关键的是前端主动链路存在高频触发风险，导致请求风暴（`/core/proactive/decision` 和 `/chat/completions` 被打满）。

### 3.2 已做抑制
- `LifeEngine` 已增加去重与冷却：
  - 首次问候门闩
  - 每日早晨问候门闩
  - life signal 冷却窗口
- 目的是避免重复触发主动决策造成连锁超时。

## 4. 当前已知瓶颈
- 回复速度较慢，主要原因：
  1. 每轮主回复都开启 `update_memory=true`
  2. 主回复目前使用全量历史（`limit=0`）
  3. 记忆提炼在主请求里同步执行（阻塞首包返回）

## 5. 下一轮优先建议
1. 主回复上下文改为窗口化（最近 N 条），不要全量历史。
2. 记忆更新改为后台异步任务（主回复先返回）。
3. 将主动调度状态机继续下沉到 Python，前端仅展示。
4. 增加错误来源标签日志（`chat_core` / `post_process` / `proxy` / `upstream`）。

## 6. 运行与验证状态
- Core 健康检查：`GET /health` 返回 200。
- Web-shell 与 Core 当前可访问（8080/8000）。
- 本地持久化文件：`data/companion.db`。
